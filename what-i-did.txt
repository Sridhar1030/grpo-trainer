vLLM Weight Sync — What Was Done (May 18, 2026)
=================================================

PROBLEM
-------
The notebook (grpo-vllm-rollout-example.ipynb) tried to do NCCL weight sync
between the workbench pod (trainer) and the vLLM pod (inference). This
deadlocked every time.

Root cause: vLLM's HTTP+NCCL weight transfer path is synchronous inside the
API server. When the trainer POSTs to /init_weight_transfer_engine, vLLM's
single worker thread blocks waiting for the NCCL group join. But the trainer
is also blocking waiting for its side of the join. When they're in separate
pods, neither can proceed — classic deadlock.

The official vLLM example (rlhf_http_nccl.py) assumes trainer and vLLM are
on the SAME machine, using localhost, on different GPUs (vLLM on GPU 0,
trainer on GPU 1).

Reference: https://docs.vllm.ai/en/v0.21.0/examples/rl/rlhf_http_nccl/


FIX
---
Matched the official example exactly:

1. Gave the vLLM deployment 2 GPUs instead of 1
2. vLLM serves on GPU 0
3. Trainer script runs INSIDE the vLLM pod on GPU 1
4. All communication happens over localhost — no cross-pod NCCL

The notebook now:
- Deploys vLLM with --load-format dummy (random weights = gibberish output)
- Writes a trainer script, copies it into the vLLM pod via oc cp
- Runs it via oc exec
- Trainer loads real Qwen2.5-0.5B weights on GPU 1
- NCCL transfers them to vLLM on GPU 0
- vLLM goes from gibberish to correct answers
